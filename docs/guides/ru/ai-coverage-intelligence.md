# Интеллектуальный анализ покрытия

Интеллектуальный анализ покрытия (Coverage Intelligence) предоставляет обзор качества тестового покрытия на уровне проекта, используя единицы покрытия на основе кода, сигналы доказательств и взвешенную оценку — без необходимости строгой трассировки требований к тестам.

## Предварительные требования

- Активна лицензия AI Pro ([см. руководство по активации](./ai-pro-license.md))
- Настроена индексация репозиториев ([см. руководство по индексации](./ai-repository-indexing.md)) — рекомендуется, но не обязательно

## Концепции

### Единицы покрытия

Вместо измерения покрытия только по количеству тестов система определяет **единицы покрытия** — значимые объекты, которые должны быть протестированы:

| Тип | Пример | Формат ключа |
|-----|--------|--------------|
| `api_endpoint` | CI Upload Allure Results | `POST:/upload/ci/allure-results` |
| `ui_flow` | Launch detail page | `launches.open-details` |
| `domain_operation` | Notification dispatch | `notifications.dispatch-on-run-complete` |

Каждая единица имеет `moduleKey` (функциональная область), флаг `isCritical` и необязательное поле `owner`.

### Сигналы доказательств

Каждая единица оценивается по 6 сигналам, нормализованным в диапазоне `[0..1]`:

| Сигнал | Что измеряет |
|--------|-------------|
| `codePresence` | Единица присутствует в текущем инвентаре кода |
| `testPresence` | К единице привязан хотя бы один автоматизированный тест |
| `executionFreshness` | Привязанные тесты были выполнены недавно |
| `executionStability` | Привязанные тесты стабильны (низкий процент нестабильности/падений) |
| `assertionDepth` | Присутствуют как позитивные, так и негативные/ошибочные проверки |
| `incidentLink` | К единице привязаны инциденты или дефекты |

### Формула оценки

Покрытие на уровне единицы:

```
effectiveCoverage = 0.25×codePresence + 0.30×testPresence
                  + 0.20×executionFreshness + 0.15×executionStability
                  + 0.10×assertionDepth
```

Оценка на уровне проекта (с учётом риска):

```
projectCoverageScore = 100 × Σ(unitCoverage × riskWeight) / Σ(riskWeight)
```

### Уверенность

Каждая единица также имеет показатель уверенности, основанный на полноте доказательств, точности привязки и актуальности данных. Он показывает, насколько можно доверять значению покрытия:

- **Высокая** (>=75%) — сильная доказательная база, надёжная оценка
- **Средняя** (50–74%) — частичные доказательства, направление верное
- **Низкая** (<50%) — ограниченные доказательства, используйте как отправную точку

## Начало работы

### 1. Откройте страницу покрытия

Перейдите в раздел **Coverage** в боковой панели (виден только с лицензией Pro) или откройте:

```
/projects/{projectId}/coverage
```

### 2. Пересоберите инвентарь

Нажмите **Rebuild Inventory**. При этом происходит следующее:

- Каталогизируются все единицы покрытия по API-поверхности платформы, UI-потокам и доменным операциям
- Запрашивается недавняя история запусков тестов для активного проекта
- Вычисляются сигналы доказательств для каждой единицы на основе эвристического сопоставления по именам тестов

### 3. Изучите дашборд

После пересборки вы увидите:

- **Coverage Score** — общий процент покрытия проекта
- **Units** — сколько единиц имеют достаточное покрытие
- **Confidence** — насколько надёжна оценка
- **Gaps** — количество единиц с недостаточным покрытием

### 4. Изучите разбивку по модулям

Раздел **Module Breakdown** показывает покрытие по функциональным областям (upload, auth, dashboard и т.д.), что позволяет определить, какие модули требуют внимания.

### 5. Просмотрите основные пробелы

Раздел **Top Coverage Gaps** отображает единицы, отсортированные по приоритету:

- **Critical** — низкое покрытие + высокий вес риска (критичная функция, нет тестов)
- **High** — ниже порога + значимый риск
- **Medium/Low** — ниже порога, меньший риск

Нажмите на пробел, чтобы развернуть и увидеть детали единицы (ID, вес риска, эффективное покрытие).

### 6. Сгенерируйте AI-рекомендации

Нажмите **Generate Recommendations**, чтобы получить рекомендации на основе AI о том, что тестировать дальше. Каждая рекомендация включает:

- **Priority** — соответствует серьёзности пробела
- **Risk reason** — почему эта единица высокорисковая
- **Coverage reason** — что отсутствует (нет тестов, слабые проверки, устаревшее выполнение)
- **Suggested scenario** — схема Given/When/Then для нового теста
- **Estimated impact** — прогнозируемое улучшение покрытия и уверенности

## Справочник API

Все эндпоинты требуют JWT-аутентификации и активной лицензии Pro.

### Пересборка инвентаря

```bash
POST /test-coverage/inventory/rebuild?projectId={id}
```

### Получение инвентаря

```bash
GET /test-coverage/inventory?projectId={id}
```

### Получение сводки покрытия

```bash
GET /test-coverage/summary?projectId={id}
```

Ответ:

```json
{
  "projectCoverageScore": 42.5,
  "totalUnits": 28,
  "coveredUnits": 12,
  "projectConfidence": 0.58,
  "confidenceBand": "medium",
  "generatedAt": "2025-06-15T10:30:00Z"
}
```

### Получение разбивки по модулям

```bash
GET /test-coverage/modules?projectId={id}
```

### Получение пробелов покрытия

```bash
GET /test-coverage/gaps?projectId={id}
```

### Генерация рекомендаций

```bash
POST /test-coverage/recommendations/generate?projectId={id}
```

### Получение сохранённых рекомендаций

```bash
GET /test-coverage/recommendations?projectId={id}
```

## Настройка весов оценки

Веса по умолчанию можно переопределить через настройки:

```bash
curl -X POST http://localhost:3001/settings/testCoverageScoringWeights \
  -H "Content-Type: application/json" \
  -b "auth_token=<jwt>" \
  -d '{
    "value": {
      "codePresence": 0.15,
      "testPresence": 0.40,
      "executionFreshness": 0.20,
      "executionStability": 0.15,
      "assertionDepth": 0.10
    }
  }'
```

После изменения весов пересоберите инвентарь и проверьте сводку — оценки будут отражать новые веса.

## Текущие ограничения

- **Единицы покрытия определяются платформой** — каталог охватывает собственную поверхность QA Report Platform (API, UI-потоки, доменные операции). Пользовательские определения единиц запланированы.
- **Привязка тестов выполняется эвристически** — тесты сопоставляются с единицами по ключевым словам в именах тестов. Явные аннотации привязки тестов к единицам запланированы.
- **Нет инкрементальных обновлений** — пересборка инвентаря заново сканирует всё. Инкрементальные обновления доказательств запланированы.
- **Рекомендации основаны на шаблонах** — при отсутствии настроенной LLM сценарии используют структурированные шаблоны. При подключённой LLM генерируются более насыщенные рекомендации на естественном языке.
